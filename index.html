ðŸ” Environment Variables

âš ï¸ Critical Security Rule

NEVER put real passwords, API keys, or secrets directly in your code. Always use environment variables. Make sure .env is listed in your .gitignore file so it's never uploaded to GitHub.

ðŸš¨ What Happens If You Accidentally Push API Keys to GitHub?

  1. Git does NOT prevent this. You must manually set up .gitignore to exclude .env files.
  2. GitHub scans for secrets. GitHub's "secret scanning" detects API key patterns (OpenAI, Anthropic, AWS, etc.) in public repos.
  3. Provider is notified. GitHub immediately alerts the LLM provider (OpenAI, Anthropic, etc.).
  4. Key is IMMEDIATELY disabled. The provider automatically disables your API key - no grace period, no warning.
  5. You get an email. You receive notification that your key was exposed and has been disabled.

âš ï¸ Key Point: The email comes AFTER your key is already disabled - not before! You don't get a chance to delete it first. Prevention (.gitignore from day one) is your only protection!

ðŸŒ¿ Branching & Small PR Changes

ðŸ’¡ Think of it like this...

Branches are like parallel universes for your code. The "main" branch is the stable, production-ready universe. When you want to try something new, you create a parallel universe (branch) where you can experiment safely. If it works, you merge it back. If it breaks, you just delete that universe â€” main stays safe.

Why Small Changes Matter:

  • Easier to review: 50-line changes are reviewable, 5000-line changes are not
  • Easier to revert: If something breaks, you know exactly what caused it
  • Faster feedback: Get reviews and merge faster
  • Less merge conflicts: Small changes rarely conflict with others
Branch Workflow
# Make sure you're on main and up to date
git checkout main
git pull origin main

# Create a new branch for your feature
git checkout -b feature/add-task-filters

# Make your changes, then stage and commit
git add .
git commit -m "Add status and priority filters to task list"

# Push your branch to GitHub
git push origin feature/add-task-filters

# Create a Pull Request on GitHub

ðŸ”¥Secret Sauce #1: The Context Window Hack

The Problem: AI can only "see" a limited amount of text at once (context window). Big projects exceed this.

The Solution: Create a PROJECT_CONTEXT.md file that summarizes your entire project. Reference it in every prompt with @PROJECT_CONTEXT.md. The AI gets the full picture without reading every file.

ðŸ”¥Secret Sauce #2: The "Act As" Technique

Instead of: "Fix this bug"

Say: "Act as a senior software engineer doing a code review. Analyze this code for bugs, performance issues, and best practice violations. Then fix what you find."

The AI gives dramatically better results when you assign it a specific expert role.

ðŸ”¥Secret Sauce #3: The Negative Prompt

Tell the AI what NOT to do. This prevents common mistakes:

"Create a contact form. DO NOT: use inline styles, hardcode colors, forget error states, skip mobile responsiveness"

ðŸ”¥Secret Sauce #4: The Rubber Duck Technique

Before asking the AI to fix something, explain the problem to it.

Say: "Let me explain what I'm trying to do..." and describe your goal, what you tried, and what went wrong. Often, just explaining it helps YOU realize the solution!

ðŸ”¥Secret Sauce #5: The Incremental Complexity Strategy

Don't ask for everything at once.

Instead of: "Build me a complete e-commerce site"

Do this: 1) Create product listing â†’ 2) Add shopping cart â†’ 3) Implement auth â†’ 4) Add checkout

Each step builds on the last. You catch errors early. Results are 10x better.

ðŸ”¥Secret Sauce #6: The Screenshot Technique

When describing UI issues, take a screenshot and describe what's wrong.

Cursor can analyze images! Paste a screenshot and say: "The button is misaligned. It should be centered. Fix it."

ðŸ”¥Secret Sauce #7: The Checkpoint Strategy

Commit working code before every major change.

Before saying "Now add authentication", commit what you have: git commit -m "Working app before adding auth"

If the next change breaks everything, you can always go back!

ðŸ”¥Secret Sauce #8: The Model Switching Strategy

Different models for different tasks:

  • cursor-small: Quick questions, syntax lookup
  • Claude Sonnet: Most coding tasks
  • Claude Opus: Complex debugging, architecture decisions
  • GPT-4o: Creative solutions, brainstorming
ðŸ”¥Secret Sauce #9: The "Show Don't Tell" Prompt

Give examples instead of explaining.

Instead of: "Make the buttons consistent"

Say: "Make all buttons look like this one: [paste the good button code]. Apply this style to: header button, form submit button, modal close button."

ðŸ”¥Secret Sauce #10: The Review Request

Ask the AI to review its own work.

After the AI generates code, say: "Review the code you just wrote. Check for: bugs, security issues, performance problems, missing edge cases. Then fix any problems you find."

The AI catches its own mistakes surprisingly often.
